## PRIMARY SUBMISSION — 45 minutes (recommended)

### Title
**Chat Is a Feature. UI Is the Product.**

### Elevator / one-liner (if a short field is offered)
Everyone is racing to "add AI" by stapling a chatbot to the corner of old software. This talk
gives you a spectrum — and a decision tool — for where AI should *actually* live in your product.

### Abstract (voting-visible)
Every team is racing to "add AI." Most of them bolt a chatbot onto the bottom-right corner of
the same software they already had — new intelligence wrapped around an old structure. The demo
dazzles. The product barely changes.

This talk makes one argument: **where you place AI is a structural product decision, not a UI
polish job.** To prove it, we take a single ordinary CRM workflow — *"prepare my Q3 follow-up
and show me what changed"* — hold the user's intent perfectly constant, and watch that same
request land in five very different places:

- **Plain chat** — fast to ship, easy to ignore.
- **Components as tools** — your design system becomes the agent's vocabulary.
- **Catalog + composition** — the agent assembles a screen from approved primitives.
- **Routing to existing tools** — sometimes the best UI is one you didn't build.
- **Shared state** — the agent stops *answering* and starts *co-editing* your product.

Same intent. Five surfaces. The interesting question is never "does the model answer?" — it's
"where does the answer live?"

Along the way we'll look at the agentic UI stack now forming (AG-UI, MCP Apps, A2UI, generative-UI
frameworks) and the real costs that arrive the moment an agent can read and mutate product state:
trust, permissions, audit, and recovery.

You'll leave with a simple decision tool you can use on Monday — a handful of questions that tell
you where a given feature belongs on the spectrum — so your next AI conversation at work is
*"this feature should be level X because…"* instead of *"should we add a chatbot?"*

Live, deterministic demos throughout. Zero vendor pitch.

### Tags / categories
`AI/ML` · `UI/UX and Design` · `Architecture and Design Patterns` · `Front End / JavaScript`

### Format
45 minutes (~40 min speaking). Talk + live demo. No audience setup required — demos are
deterministic and run locally, so they work without network or API keys.

## ALTERNATE SUBMISSION — 20 minutes (your 2nd allowed talk, or a fallback)

### Title
**Stop Bolting Chatbots onto Old Software**

### Abstract (voting-visible)
The default way to "add AI" to a product is to staple a chatbot to the corner of software that
was designed for a pre-AI world. It demos beautifully and changes nothing.

In 20 minutes and one live CRM workflow, I'll send the *same* user request to five different
places — plain chat, components, composed layouts, routed tools, and shared state — and make the
case that **where intelligence lives is a product decision, not a styling choice.**

You'll leave with a four-question decision tool for placing AI in your own product, and one
sentence to try in your next standup: *"this feature should be level X, because…"*

### Tags / categories
`AI/ML` · `UI/UX and Design` · `Front End / JavaScript`

### Format
20 minutes (~15 min speaking). Talk + one or two tight live demos.
